**Why a merge is not viable.** Upstream rewrote the exact core files these customizations
target. A dry-run `git merge upstream/main` produces **38 conflicts** in the worst files
(`index.ts`, `config.ts`, `db.ts`, `container-runner.ts`, `channels/registry.ts`, `ipc.ts`,
`router.ts`, `types.ts`). Upstream also **restructured** the tree: `src/db.ts` → `src/db/`
directory (sessions split into `src/db/sessions.ts`), and a whole new `src/cli/` tree was
added. So a future upgrade is an intent **replay**, not a merge — which is what these docs
enable.

## How to use this for a future upgrade

1. Read [`UPGRADE-PLAYBOOK.md`](UPGRADE-PLAYBOOK.md) — the deferred Phase-2 procedure
   (worktree on clean upstream → reapply skills → replay subsystems in priority order →
   validate → swap).
2. Work the subsystems in **priority order** (below), highest-risk first.
3. For each subsystem, open its file under `subsystems/` — each has Intent, Files,
   reproduction detail, and risk notes.
4. Cross-check every site-specific value against [`infra/site-constants.md`](infra/site-constants.md).

## Migration priority order (highest conflict risk first)

1. **Credential proxy** (subsystem 1) — must precede all container work; replaces OneCLI entirely.
2. **Session lifecycle + MCP hash** (subsystem 3) — DB schema + `runAgent()` rewrite; collides with upstream `src/db/` split.
3. **Per-container credential injection** (subsystem 1) — `buildContainerArgs()` is unrecognizable vs upstream.
4. **Volume mounts + gated skills** (subsystem 1/6) — `buildVolumeMounts()` evolves upstream.
5. **Container recycle on hash drift** (subsystem 3) — async probe in the message hot-path.
6. **Container image** (subsystem 6) — Dockerfile a-mem/Python layers; agent-runner MCP registry.
7. **Outbound formatting** (subsystem 5) — `formatOutbound()` signature change propagates everywhere.
8. **Obsidian settings** (subsystem 2) — replaces constant exports with lazy-read functions.
9. Everything else (LOW risk) — apply after the high-risk items stabilize.
