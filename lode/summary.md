# NanoClaw — Summary

Personal Claude assistant built as a single Node.js orchestrator (`src/index.ts`) with a skill-based channel system. Channels (WhatsApp, Telegram, Slack, Discord, Gmail, Protonmail) self-register at startup and route inbound messages to a Claude Agent SDK instance running inside a per-group container (Linux VM) with an isolated filesystem and memory at `groups/{name}/`. The OneCLI gateway (port 3002) injects secrets at request time so containers never see raw keys. Scheduled tasks run via `src/task-scheduler.ts`; IPC between host and containers flows through `src/ipc.ts`. Email ingestion runs in a separate project at `~/Projects/mailroom/` (`gitlab.com/jmcstone/mailroom`) — its own GitLab repo as of 2026-04-25. The compose stack at `~/Projects/ConfigFiles/containers/mailroom/mailroom/` pulls `mailroom-local:latest` (built from the source repo) and runs four services: the ingestor polls Gmail + Proton IMAP, a rules engine classifies + labels + archives at ingest time, and events (`inbox:urgent` / `inbox:routine` / silent) push into nanoclaw's `mailroom-subscriber` channel; see `lode/infrastructure/madison-pipeline.md` here for nanoclaw's subscriber path. Mailroom-side architecture (sync workers, mirror data model, rule engine, schema) is documented in `~/Projects/mailroom/lode/`. The store is a full upstream mirror (labels, folder membership, direction, archived/deleted flags) kept current by Gmail `history.list` incremental sync, Proton IMAP IDLE on INBOX plus per-folder UIDNEXT polling, and a three-tier recency reconcile (hot 30 min / warm 6 h / cold weekly). Active Jeff-specific deployment: Telegram + Gmail + Protonmail channels registered, agent persona "Madison", persistent data under `~/containers/data/NanoClaw` and `~/containers/data/mailroom` (BTRFS subvolume, auto-snapshotted; see `lode/infrastructure/persistence.md`), downloads to `/workspace/downloads` inside containers. Each working group has an isolated Obsidian vault, a per-group a-mem (agentic memory) ChromaDB, and a per-group context-mode FTS5 knowledge base; all baked into the container image. See `lode/infrastructure/a-mem.md` and `lode/infrastructure/context-mode.md`.
