# NanoClaw — Summary

Personal Claude assistant built as a single Node.js orchestrator (`src/index.ts`) with a skill-based channel system. Channels (WhatsApp, Telegram, Slack, Discord, Gmail, Protonmail) self-register at startup and route inbound messages to a Claude Agent SDK instance running inside a per-group container (Linux VM) with an isolated filesystem and memory at `groups/{name}/`. The OneCLI gateway (port 3002) injects secrets at request time so containers never see raw keys. Scheduled tasks run via `src/task-scheduler.ts`; IPC between host and containers flows through `src/ipc.ts`. Email ingestion runs in a separate Docker stack at `~/Projects/ConfigFiles/containers/mailroom/`: the ingestor polls Gmail + Proton IMAP, a rules engine classifies + labels + archives at ingest time, and events (`inbox:urgent` / `inbox:routine` / silent) push into nanoclaw's `mailroom-subscriber`; see `lode/infrastructure/mailroom-rules.md` + `lode/infrastructure/madison-pipeline.md`. The store is a full upstream mirror (labels, folder membership, direction, archived/deleted flags) kept current by Gmail `history.list` incremental sync and Proton IMAP IDLE+CONDSTORE with nightly reconcile; see `lode/infrastructure/mailroom-mirror.md` + `lode/architecture/madison-pipeline.md`. Active Jeff-specific deployment: Telegram + Gmail + Protonmail channels registered, agent persona "Madison", persistent data under `~/containers/data/NanoClaw` and `~/containers/data/mailroom` (BTRFS subvolume, auto-snapshotted; see `lode/infrastructure/persistence.md`), downloads to `/workspace/downloads` inside containers. Each working group has an isolated Obsidian vault, a per-group a-mem (agentic memory) ChromaDB, and a per-group context-mode FTS5 knowledge base; all baked into the container image. See `lode/infrastructure/a-mem.md` and `lode/infrastructure/context-mode.md`.
